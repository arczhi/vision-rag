Vision RAG 生产化改造蓝图

目标：可靠的视频高光标注与视频入知识库能力，面向 Ubuntu + NVIDIA GPU 的夜间顺序批处理。

改造原则

当前实现已经具备视频入库、Qdrant named vectors、知识库检索和高光标注能力。生产化的关键是把“长耗时模型调用”从 API 线程和内存任务里抽出来，变成可恢复、可追踪、可限流的 durable workflow。

API只接收请求、校验参数、创建 durable job。
Scheduler每天固定时间生成 batch run，按优先级派发。
GPU Worker单卡串行或低并发执行 ASR、VLM、embedding。
StoresPostgres 保存状态；Qdrant 保存向量；对象存储保存视频和截图。
Review UI查看结果、修正、接受/拒绝、回写反馈。

第一优先级

把 `TaskManager` 从 in-memory 替换为持久化 job 表。
API 与 GPU worker 分进程部署，API 不加载大模型。
每个视频一个可重试 workflow，步骤输出可审计。
夜间调度只生产队列，不在 request path 上做重活。
1 GPU worker 并发默认值。单卡先串行，稳定后再试探并发。
避免显存抖动
5-10m 单条视频长度。适合按 clip 粒度 checkpoint，失败后从步骤继续。
clip / segment checkpoint
22:00 推荐夜间批处理窗口起点。下一次运行时间必须持久化展示。
调度是产品行为
3 核心状态源：Postgres、Qdrant、对象存储。不要再把任务结果只放内存。
内存任务不可恢复

为什么不用 API 进程跑批量任务

重启会丢任务

当前 `api/tasks.py` 明确是 in-memory 单进程任务管理。生产中 API 重启、发布、崩溃都会让 `running` 任务和结果不可恢复。

显存资源需要全局排队

5-10 分钟视频会触发解码、ASR、embedding、VLM 多段推理。需要跨请求、跨批次的 GPU 锁和显存预算，而不是每个请求自己抢。

调度可见性不足

运营需要知道今晚跑哪些、下次什么时候跑、哪条卡住、失败能否重试。进程健康不等于工作流健康。

夜间批处理节奏

假设每天 22:00 开始，单卡顺序执行。调度器只负责生成 batch run，worker 根据资源顺序消费。

21:55冻结待处理队列，计算今晚 capacity。
22:00创建 `batch_run` 和每条视频的 `workflow_run`。
22:05GPU worker 预加载模型并开始第一条视频。
整晚按视频顺序执行；失败视频进入 retry 或 failed，不阻塞后续。
07:30停止领取新任务；已运行步骤可完成或软暂停。
08:00生成日报：成功、失败、待复核、GPU 时长、错误 TopN。

单条视频工作流

ingest.prepare校验文件、探测时长、生成 video_id、去重。
clip.index切片、抽帧、缩略图、ASR、caption embedding。
highlight.recall用 visual/caption/transcript 查 KB，聚合候选。
highlight.judgeQwen2-VL 只判断候选，不全量扫所有 clip。
review.ready结果入库，等待人工接受/拒绝或自动发布。
显卡受限策略：embedding 和 VLM 阶段共享同一 GPU semaphore；优先跑候选少、时长短的视频；每步记录显存峰值和耗时。先把 `concurrency=1` 做稳，再考虑 ASR CPU 并行、GPU 推理串行。

状态机

状态含义可转移到可见性要求
queued已排队，未被 worker 领取running, canceled显示排队位置、预计开始时间
runningworker 已锁定并执行succeeded, failed, retry_wait, paused显示 worker_id、locked_at、current_step
retry_wait失败后等待重试queued, failed显示 attempts、next_retry_at、error_code
succeeded_empty流程成功但无高光needs_review, archived不要误判为失败，保留阈值和召回统计
needs_review结果需要人工确认approved, rejected, rerun_requested展示 evidence、score_breakdown、模型版本
failed不可恢复或超过重试queued, archived机器错误码 + 用户可读说明 + trace_id

结果可复现字段

video_checksum clip_config_hash embedding_model_version vlm_model_version kb_version scoring_config prompt_version trace_id

AI 判断不是纯确定性行为，必须保存输入事实、模型版本、prompt 版本、输出 JSON、人工反馈，才能解释“为什么这段被选中”。

健康检查分层

检查回答的问题失败例子
/health/live进程是否活着事件循环挂死、进程退出
/health/ready依赖是否可用Postgres/Qdrant 不通
/ops/workflows工作流是否推进queue depth 增长、running 超时
/ops/gpuGPU worker 是否正常显存不足、模型未加载、心跳丢失

分阶段改造路线

第 1 周：把任务状态落库新增 Postgres、`workflow_runs`、`workflow_steps`，保留现有 API 但不再只依赖内存任务。先让 `/annotate` 和 `/ingest` 写 durable run。
第 2 周：抽出 workflow service把 `api/server.py` 中的闭包任务迁移到 `services/ingest_workflow.py`、`services/annotation_workflow.py`，API 只创建 run。
验收标准重启 API 后，任务仍可在 UI 查询；失败有 `error_code`、`error_detail`、`current_step`；重复提交同一视频不会生成重复 active run。
第 3 周：scheduler + worker实现 `workers.scheduler` 和 `workers.gpu_worker`，每天 22:00 创建 batch，worker 用 `SKIP LOCKED` 顺序领取。
第 4 周：GPU 资源治理配置 `GPU_WORKER_CONCURRENCY=1`、步骤超时、stale lock 回收、失败重试退避、最大夜间窗口。
验收标准一批 10 条 5-10 分钟视频可以无人值守跑完；中途 kill worker，重启后从未完成步骤继续。
第 5 周：标注质量仪表盘新增空结果率、人工接受率、label 分布、score_breakdown 分布、模型版本对比。
第 6 周：评测集沉淀 50-100 条人工确认视频，形成 nightly 或 weekly eval，防止阈值/模型/KB 改动退化。
验收标准每次调阈值、模型、KB 后都能看到召回率、接受率、平均片段时长和误报样例。
多卡拆分每张卡一个 worker 进程；按 project/kb/video 优先级分配；不同模型可绑定不同 GPU。
吞吐优化ASR 可 CPU 并行，embedding 批量化，VLM 只对 recall 后候选执行；对象存储与 Qdrant 写入异步批处理。
验收标准增加 GPU 后 throughput 近似线性增长，且单视频结果与单卡串行模式一致。

不建议先做的事

不要先换更大的 VLM。当前瓶颈是任务可靠性和可观测性，不是单点模型能力。
不要把 cron 直接 curl `/annotate`。这仍然会把长任务绑在 API 生命周期上。
不要用“进程健康”当作“今晚批次成功”。必须查 durable run 和步骤状态。
不要让失败静默 fallback。fallback 也要写入状态和错误码。
