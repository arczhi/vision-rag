# Qwen3.5-Plus 高光推理「时间戳幻觉」优化方案

> 面向：入库链路开发同学 / 高光产品同学
> 状态：已落地，已用 3 个真实问题视频验证收敛
> 涉及文件：`ingest/llm_segments.py`

---

## 一、问题是什么（产品视角）

用户提交一批视频做「高光自动切分」，部分视频入库后**一个高光片段都没有**，前端显示「LLM 无 segments，走默认切片」。

排查 9 个真实任务后定位到一类核心故障：**模型把短视频的高光时间点，编造成了几十分钟。**

举例（真实数据）：

| 视频 | 真实时长 | 模型输出的最大时间戳 | 结果 |
|------|---------|-------------------|------|
| 血染茉莉花ep1 | **103 秒**（1.7 分钟） | **5880 秒**（98 分钟） | 段全部作废 |
| 2月4日(1) | **138 秒**（2.3 分钟） | **8040 秒**（134 分钟） | 段全部作废 |
| 2月4日(7) | **84 秒**（1.4 分钟） | **4860 秒**（81 分钟） | 段全部作废 |

模型输出的时间戳还高度规整（`00:02:00`、`00:14:00`、`00:26:00`…几乎等距 9~12 分钟），一看就不是真实剧情节奏，而是模型在「按套路硬编」——这种有规律的瞎编，比随机出错更说明它根本没看懂视频时长。

---

## 二、为什么会幻觉（开发视角 · 底层逻辑）

根因不在代码兜底，在**我们让模型输出的格式本身**。

老方案让模型直接输出 `"timestamp": "HH:MM:SS"` 字符串。这等于把两个负担同时压给模型：

1. **格式正确性**：`HH:MM:SS` 有 6 个字段位，模型要保证 `时:分:秒` 都合法（分秒不能 >59）。
2. **数值合法性**：时间点必须落在视频真实时长内。

而模型并不是把视频「逐帧完整看完」的——为了省算力，它每秒只抽 2 帧画面来看（相当于把视频压成一份很稀疏的幻灯片）。画面信息这么稀，模型对「这个视频到底多长、节奏多快」就缺乏稳定感知，于是退化成「按固定间隔编一串看起来合理的时间戳」。

下游解析代码 `_segments_from_json` 拿到这些超长时间戳后：
- 每段 `end = min(下一段start, 时长)`，当 `start` 已经远超真实时长，`end` 被钳到时长 → `end <= start` → 段被判定无效丢弃 → **kept=0**。

**一句话：让模型一次要操心的事太多 → 它扛不住就开始瞎编 → 下游被迫把这些段全丢。**

## 三、怎么修（顶层设计 · 两层防御）

核心思路：**让模型少操心 + 代码侧再加一道硬兜底**。提示词负责把质量做上去，代码负责保证结果一定合法，两层缺一不可。

### 第一层：让模型输出的时间格式，从「字符串时间戳」改为「整数秒区间」（治本）

模型不再拼 `HH:MM:SS`，直接输出整数秒的 `start` / `end`：

**改之前：**
```json
{ "highlights": [ { "timestamp": "00:00:10", "description": "..." } ] }
```

**改之后：**
```json
{ "highlights": [ { "start": 10, "end": 16, "description": "..." } ] }
```

为什么更稳：
- **无格式可错**：整数 `16` 不存在「分钟>59」这类格式陷阱。
- **天然成段**：`[start, end]` 直接是片段，不再靠「下一段反推 end」的脆弱逻辑。
- **越界一眼可判**：`0 <= start < end <= 时长`，一行代码校验。

代码落点：`ingest/llm_segments.py:132` `DEFAULT_USER_PROMPT_TEMPLATE` 的 Output Format 与 Rules。

### 第二层：prompt 内注入「真实时长上限」（强约束）

每次推理前，把视频真实时长动态写进 prompt：

```
## 重要时长约束
本视频总时长仅为 103 秒（约 01:43）。
所有 start 和 end 都是【整数秒】，必须满足 0 <= start < end <= 103。
严禁输出大于 103 的数值，严禁把短视频的时间点编造成几十分钟。
```

代码落点：`ingest/llm_segments.py:397` `call_qwen_video()`，新增 `video_duration` 参数。

### 第三层：代码侧硬兜底（万一模型仍不听话）

即使 prompt 失效，解析层兜底保证不污染数据：

- `start >= 时长` 的段 → 判定为幻觉，**丢弃**，并累计 `dropped_oob` 计数 + 日志告警；
- `end` 超过时长 → **自动截到视频结尾**；
- 单段超 8 秒 → 截到 8 秒；
- 向后兼容：仍能解析老的 `timestamp` 字段，老数据不受影响。

代码落点：`ingest/llm_segments.py:300` `_extract_start_end()` + `:317` `_segments_from_json()`。

## 四、效果证明（已验证 · 闭环）

| 视频 | 真实时长 | 解析段数 | **保留 kept** | **越界丢弃 oob** | **最大时间戳** |
|------|---------|---------|--------------|-----------------|---------------|
| 血染茉莉花ep1 | 103s | 17 | **17** | **0** | 103s ✓ |
| 2月4日(1) | 138s | 13 | **13** | **0** | 138s ✓ |
| 2月4日(7) | 84s | 11 | **11** | **0** | 84s ✓ |

### 修复前 vs 修复后对比

| 指标 | 修复前 | 修复后 |
|------|--------|--------|
| 最大时间戳 | 5880 / 8040 / 4860 秒（幻觉到 80~134 分钟） | 103 / 138 / 84 秒（精确落在时长内） |
| 保留段数 kept | **0 / 0 / 0**（全军覆没） | **17 / 13 / 11**（全部保留） |
| 越界丢弃 oob | 16 / 14 / 13（全部越界） | **0 / 0 / 0**（零越界） |

## 六、可观测性（产品可看监控指标）

每个入库任务的 `task.extra.llm_segment_stats` 现在回填：

| 字段 | 含义 |
|------|------|
| `parsed_total` | 模型解析出的总段数 |
| `kept` | 实际保留的有效段数 |
| `dropped_oob` | 因越界（幻觉）丢弃的段数 |
| `max_timestamp` | 模型输出的最大时间戳（用于判断是否仍在幻觉） |
| `video_duration` | 视频真实时长 |

**建议监控抓手**：若线上 `dropped_oob / parsed_total` 比例升高，说明模型幻觉抬头，需复查 prompt 或模型版本。

## 七、遗留与边界

- **prompt 是软约束**：模型不保证 100% 听话，所以第三层代码兜底必须保留，不能删。
- **真·长视频场景**：本方案针对「短视频被瞎编成长视频」。若未来要支持 30 分钟+ 真长视频，需另开任务（让模型每秒多抽几帧画面、或把长视频拆段分别推理），不在本次范围。
- **time-axis 单位**：prompt 已显式声明「整数秒，从第 0 秒起算」，规避模型误解为帧号/百分比。
