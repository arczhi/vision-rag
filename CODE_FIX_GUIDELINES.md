# 代码修改指南: 不要只补一个错误点

这个文件用于提醒后续改代码时的默认工作方式。用户指出某个地方做错了，不要只在原逻辑上加一个针对该错误的特殊判断。那通常只能避开当前样例，不能解决同类型问题。

## 核心原则

1. 先确认根因，再改代码。
   - 找出错误是由数据结构、职责边界、提示词、状态流、评分逻辑、缓存、排序还是 UI 展示造成的。
   - 不要把用户举的例子当成唯一问题。

2. 避免特化补丁。
   - 不要写只针对某个关键词、某个样例、某个文件名、某个标签、某个时间段的逻辑。
   - 如果必须处理例外，要说明为什么这是业务规则，而不是临时绕过。

3. 抽象成同类问题的通用约束。
   - 把“这次错在哪里”提升成“这类错误为什么会发生”。
   - 修复点应该能覆盖未来相同模式的问题。

4. 保持原有架构清晰。
   - 不要在调用层堆 if/else 掩盖下游职责错误。
   - 应该在最接近根因的位置修复，比如数据来源混淆就重构上下文结构，排序语义错误就修排序规则。

5. 验证要覆盖同类场景。
   - 除了用户指出的样例，还要检查至少一个相邻或相反场景。
   - 如果无法跑完整验证，要说明已验证什么、还缺什么。

## 修复前检查

- 这个错误是否来自错误的数据被传入？
- 是否把参考信息当成当前事实？
- 是否有旧逻辑的字段命名误导模型或调用者？
- 是否有排序、过滤、fallback 让错误结果重新浮上来？
- 是否有缓存或任务恢复导致页面看到旧结果？
- 这个修复是否只对用户提到的词或样例生效？

## 推荐做法

- 先重命名或拆分容易混淆的数据结构。
- 在接口边界明确字段语义，例如 current、reference、fallback、rejected。
- 用通用规则替代样例枚举。
- 把拒绝、未检查、fallback 结果分组展示，不让它们和确认结果混排。
- 在 prompt 或注释里描述职责和优先级，不写针对单个失败样例的规避语句。

## 不推荐做法

- 用户说“龙喷火不对”，就写“如果样例有龙喷火但当前没有就不要输出龙喷火”。
- 用户说“苹果不相关”，就给“苹果”加同义词或黑名单。
- 用户说“只返回 8 条”，就把 8 改成 20，而不检查候选生成、过滤、排序和拒绝逻辑。
- 用户说“刷新丢失”，就只把当前页面状态塞进 localStorage，而不看任务结果是否可按 ID 恢复。

## 输出前自检

完成修改后，回答用户前先确认:

- 修复点是否在根因位置？
- 是否移除了样例级特殊逻辑？
- 是否保留了同类问题的通用防护？
- 是否跑了最相关的编译、单元、接口或页面验证？
- 是否清楚说明还有哪些旧任务结果需要重新生成才会看到新逻辑？
